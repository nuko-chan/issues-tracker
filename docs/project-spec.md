# Issue Tracker API — プロジェクト仕様書

> **このドキュメントの目的**: 2 週間でバックエンドポートフォリオを完成させるための「完了基準」と「判断ガイド」。
> AI ツール（Claude Code / Cursor 等）にこのファイルを常に読み込ませて、ブレずに開発を進めること。

---

## 🎯 このプロジェクトのゴール（最重要）

**面談で「バックエンドをゼロから設計・構築した経験」を技術的に語れるようにすること。**

つまり：

- 「なぜその設計にしたか」を自分の言葉で説明できる
- 「他の選択肢と比較して何を捨てたか」を語れる
- 「実装で詰まったポイントとどう解決したか」を語れる

**ゴールは「すごいアプリを作ること」ではない。「設計判断を語れる成果物を持つこと」。**

---

## 📋 作るもの

GitHub Issues / GitHub Projects の簡易クローン。
マルチユーザー対応の Issue トラッキング API + 最小限のフロントエンド。

### ドメインモデル（概要）

```
User ──< TeamMember >── Team
                          │
                       Project
                          │
                        Issue ──< Comment
                          │
                        Label ──< IssueLabel
```

- **User**: 認証の主体。メールアドレス + パスワードでサインアップ/ログイン
- **Team**: ユーザーのグループ。オーナーとメンバーのロールを持つ
- **Project**: Team に属する。Issue を束ねる単位
- **Issue**: タイトル / 本文 / ステータス（open, in_progress, closed）/ 優先度 / 担当者
- **Comment**: Issue に対するコメント
- **Label**: Project 内で定義されるラベル。Issue に複数つけられる（多対多）

## 🛠 技術スタック

| レイヤー   | 技術                              | 選定理由（面談で語る）                                   |
| ---------- | --------------------------------- | -------------------------------------------------------- |
| Runtime    | Node.js + TypeScript              | 案件の技術スタック要件に直結                             |
| Framework  | Express                           | 案件採用率が高い。ミドルウェアの仕組みを素で理解するため |
| ORM        | Prisma                            | 型安全・マイグレーション管理・直感的なAPI                |
| DB         | PostgreSQL (Docker)               | 実務で使用経験あり。リレーション設計の表現力             |
| Validation | Zod                               | 既に実務で習熟済み。FE/BE 共通スキーマの可能性を語れる   |
| Auth       | JWT (jsonwebtoken) + bcrypt       | passport.js を使わず仕組みを理解する                     |
| Test       | Vitest + supertest                | 既に実務で習熟済み。API 統合テストを書く                 |
| Frontend   | React + TanStack Query + Tailwind | 得意領域。最小限でよい                                   |
| Infra      | Docker Compose                    | DB + API + Frontend を 1 コマンドで起動                  |

### フェーズ 1: 設計（必須）

> 手を動かす前に最低限これだけ決める。完璧にしない。A4 一枚分くらいの粒度でいい。

- [ ] **ER 図**を mermaid 記法で作成（README.md に記載）

- [ ] **エラーレスポンスの統一フォーマット**を定義
  - 例: `{ error: { code: "VALIDATION_ERROR", message: "...", details: [...] } }`
  - HTTP ステータスコードの使い分け方針（400 / 401 / 403 / 404 / 409 / 422 / 500）
- [ ] **認証フローの概要**を図示（signup → login → JWT 発行 → リクエスト時の検証）
- [ ] **ディレクトリ構成**を決定（レイヤードアーキテクチャ）

### フェーズ 2: 認証基盤（最重要 — ここが一番の学び）

> このフェーズが面談で最も聞かれる。理解を優先し、自分の手で書く。

- [ ] **User テーブル + Prisma マイグレーション**が動く
- [ ] **パスワードのハッシュ化**（bcrypt）が実装されている
  - 平文パスワードが DB に保存されていないことを確認
- [ ] **サインアップ API** (`POST /api/auth/signup`)
  - Zod でバリデーション（メール形式・パスワード強度）
  - 重複メールアドレスの 409 Conflict ハンドリング
- [ ] **ログイン API** (`POST /api/auth/login`)
  - bcrypt.compare でパスワード照合
  - JWT トークン発行（有効期限付き）
- [ ] **認証ミドルウェア**が動く
  - Authorization ヘッダーから JWT を検証
  - `req.user` にユーザー情報をセット
  - 無効/期限切れトークンの 401 レスポンス
- [ ] **エラーハンドリングミドルウェア**が動く
  - 全 API のエラーを一元処理
  - 本番では stack trace を返さない
  - Zod のバリデーションエラーを統一フォーマットに変換
- [ ] **リクエストログミドルウェア**が動く（簡易でよい。method / path / status / duration）

### フェーズ 3: CRUD エンドポイント（実力が活きるフェーズ）

> ここは既存スキルの延長。速度を出す。

**Team**

- [ ] `POST /api/teams` — チーム作成（作成者が自動的にオーナーになる）
- [ ] `GET /api/teams` — 自分が所属するチーム一覧
- [ ] `POST /api/teams/:id/members` — メンバー招待（オーナーのみ）
- [ ] `DELETE /api/teams/:id/members/:userId` — メンバー削除（オーナーのみ）

**Project**

- [ ] `POST /api/teams/:teamId/projects` — プロジェクト作成
- [ ] `GET /api/teams/:teamId/projects` — プロジェクト一覧

**Issue**

- [ ] `POST /api/projects/:projectId/issues` — Issue 作成
- [ ] `GET /api/projects/:projectId/issues` — Issue 一覧（以下対応）
  - [ ] ページネーション（cursor-based または offset）
  - [ ] ステータスでフィルタ
  - [ ] 優先度でソート
- [ ] `GET /api/projects/:projectId/issues/:issueId` — Issue 詳細
- [ ] `PATCH /api/projects/:projectId/issues/:issueId` — Issue 更新（ステータス変更含む）
- [ ] `DELETE /api/projects/:projectId/issues/:issueId` — Issue 削除

**Comment**

- [ ] `POST /api/issues/:issueId/comments` — コメント追加
- [ ] `GET /api/issues/:issueId/comments` — コメント一覧
- [ ] `DELETE /api/comments/:commentId` — コメント削除（本人のみ）

**Label**

- [ ] `POST /api/projects/:projectId/labels` — ラベル作成
- [ ] `PUT /api/issues/:issueId/labels` — Issue へのラベル付け替え（多対多）

### フェーズ 4: 認可（ロールベースアクセス制御）

> 認証（誰か）と認可（何ができるか）を分離して語れることが面談で効く。

- [ ] **認可ミドルウェア**が動く
  - Team オーナー: メンバー管理・プロジェクト削除が可能
  - Team メンバー: Issue / Comment の CRUD が可能
  - 非メンバー: 403 Forbidden
- [ ] **リソースの所有権チェック**
  - Comment の削除は作成者本人のみ
  - Issue の削除はオーナーまたは作成者のみ

### フェーズ 5: テスト（設計の穴を見つけるフェーズ）

> 全部テストする必要はない。「テスト戦略を語れること」が大事。

- [ ] **認証系のテスト**（最優先）

- [ ] **認可系のテスト**

  - 非メンバーのアクセス: 403
- [ ] **Issue CRUD のテスト**（最低限）

- [ ] **テスト用 DB のセットアップ/ティアダウン**が動く

## ⚠️ 開発ルール（完璧主義を殺すためのルール）

### ルール 1: 3 時間ルール

**1 つのトピックに 3 時間以上ハマったら、TODO コメントを残して次へ進む。**

```typescript
// TODO: トランザクションの分離レベルを調べる。READ COMMITTED で問題ないか要確認
// 一旦デフォルトで進める（2026-02-20）
```

3 時間以上かけて「理解してから次へ」は禁止。動くコードを先に書き、理解は後から言語化する。

### ルール 2: 設計は「決める」もの、「正解を見つける」ものではない

バックエンドの設計に唯一の正解はない。「A と B を比較して、このプロジェクトでは A を選んだ。理由は ○○」が語れればそれでいい。迷ったら DECISIONS.md に書いて次へ進む。

```markdown
## JWT の保存場所

### 選択肢

- A: localStorage → XSS に弱い
- B: httpOnly Cookie → CSRF 対策が必要
- C: メモリ（変数）→ リロードで消える

### 判断

このプロジェクトではCを採用。理由: ポートフォリオの主目的はAPIの設計であり、
トークン管理のセキュリティ議論は面談で「選択肢と tradeoff を知っている」ことを
示せれば十分。本番ではBが望ましい。
```

### ルール 3: コミットは小さく、頻繁に

「完成してからコミット」は禁止。以下の粒度：

- 「User モデルと マイグレーション追加」
- 「サインアップ API の実装」
- 「認証ミドルウェアの追加」
- 「Issue CRUD の Controller 実装」

コミット履歴そのものがポートフォリオになる。面談官は README だけでなくコミット履歴も見る（推測だが、技術力を見る人はそうする傾向がある）。

### ルール 4: フロントに逃げない

詰まったときに「フロントだけ先に作ろう」と逃げるのは禁止。
このプロジェクトの主役はバックエンド。フロントは Day 12-13 まで触らない。
API は Postman / curl / Vitest で動作確認する。

### ルール 5: AI の使い方

| 使ってよい                 | 使い方に注意                                              |
| -------------------------- | --------------------------------------------------------- |
| ボイラープレートコード生成 | 生成されたコードの「なぜ」を説明できるか確認              |
| エラーメッセージの調査     | そのまま貼り付けて解決するのは OK                         |
| 設計の壁打ち               | AI の提案を鵜呑みにせず、自分の判断を DECISIONS.md に残す |
| テストケースの洗い出し     | AI にエッジケースを聞くのは既に得意なはず                 |

**認証ミドルウェアとエラーハンドリングミドルウェアだけは、コード生成に頼らず自分の手で書く。**
この 2 つがバックエンドの「考え方」の核。Express のリクエスト処理パイプラインを理解するための最短経路。

### レイヤーの責務（面談で語るポイント）

| レイヤー       | 責務                                                                | 依存方向             |
| -------------- | ------------------------------------------------------------------- | -------------------- |
| **Route**      | URL とコントローラーの紐付け                                        | → Controller         |
| **Controller** | HTTP リクエストの受け取り・レスポンスの返却。バリデーション呼び出し | → Service, Validator |
| **Service**    | ビジネスロジック。DB 操作の組み立て。トランザクション管理           | → Prisma Client      |
| **Validator**  | Zod スキーマ定義。入力の型保証                                      | なし（純粋な定義）   |
| **Middleware** | 横断的関心事（認証・認可・エラー処理・ログ）                        | → Service（認可時）  |

**Controller に直接 Prisma を書かない。** Service 層を経由させる。
理由: テスタビリティの確保と、ビジネスロジックの所在を明確にするため。
面談で「なぜ分けたか」を聞かれたときに「Controller はリクエストの入口と出口に責任を持ち、Service はビジネスルールに責任を持つ。テストでは Service 層を単体テストできる」と説明する。
